# Purpose

Marketing Hero Reviews Widget is a lightweight embeddable reviews platform that allows businesses to display public Google reviews on any website using a single script tag.

The system exists to provide fast, attractive, configurable review displays for Marketing Hero / Adriel Partners client sites without requiring WordPress plugins, heavy JavaScript frameworks, or per-site custom code.

# 2. System Type

## Classification

Cloudflare-hosted embeddable widget platform.

This is not a WordPress plugin, not a traditional Nuxt app, and not a full SaaS dashboard in v1.

## Why This Classification Is Correct

The product runtime is a Cloudflare Worker that serves:

- widget JavaScript
- CSS/assets if needed
- review API responses
- admin-protected refresh endpoint
- scheduled sync jobs

The widget runs on third-party/client websites through a script tag.

The project uses Cloudflare Worker + KV + scheduled triggers rather than a VPS-hosted server and PostgreSQL database in v1.

# 3. Product Scope

## Version One Goals

- Serve a single embeddable widget script from `reviews.marketinghero.net`.
- Support multi-tenant usage from day one through multiple Google Place IDs.
- Allow configuration through `data-*` attributes on the script tag.
- Support flyout and inline display modes.
- Support inline grid and carousel layouts.
- Support built-in themes.
- Support optional custom CSS overrides.
- Fetch Google Places review data through a scheduled refresh model.
- Refresh registered active places every 24 hours.
- Store normalized review data in Cloudflare KV.
- Serve widget review data from KV only during normal page loads.
- Provide an admin-protected manual refresh endpoint.
- Keep the runtime fast, dependency-light, and portable within Cloudflare.

## Explicit Non-Goals for v1

- No customer-facing admin dashboard.
- No user account system.
- No billing system.
- No self-service onboarding.
- No client-managed API keys in v1.
- No Google OAuth in v1.
- No Google Business Profile API integration in v1.
- No durable relational database in v1.
- No WordPress plugin requirement.
- No heavy frontend framework.
- No npm dependency stack unless clearly justified.

## Future SaaS Goals

The architecture should leave room for:

- customer accounts
- widget profiles saved in a database
- dashboard-based configuration
- encrypted per-client Google Places API keys
- Google Business Profile OAuth
- billing/subscriptions
- usage limits
- widget analytics
- agency/team management
- multiple review source adapters

Future SaaS expansion should not be built prematurely in v1.

# 5. Hosting and Portability

## Hosting Model

Production runtime:

```text
Client Website
→ <script src="https://reviews.marketinghero.net/widget.js" ...>
→ Cloudflare Worker
→ Cloudflare KV
```

Scheduled refresh:

```text
Cloudflare Cron Trigger
→ Cloudflare Worker scheduled handler
→ Google Places API
→ Normalize review data
→ Cloudflare KV
```

Manual refresh:

```text
Admin request
→ POST /api/admin/refresh-place
→ Bearer token verification
→ Google Places API
→ Normalize review data
→ Cloudflare KV
```

## Portability Requirement

The project is intentionally Cloudflare-native in v1.

Portability means:

- source code stays clear and dependency-light
- Google API logic is isolated in an adapter/service
- KV access is isolated behind storage functions
- widget rendering is not coupled to Worker internals
- future migration to another edge/server platform remains possible with reasonable effort

## Infrastructure Constraints

Cloudflare Workers impose constraints around:

- request CPU time
- scheduled execution time
- KV eventual consistency
- secret storage
- environment bindings
- response caching

Agents must respect these constraints and avoid building long synchronous request flows.

Represents a Google business/place being tracked.

Fields may include:

- `placeId`
- `businessName`
- `active`
- `createdAt`
- `lastRefreshAt`
- `lastRefreshStatus`
- `source`

## Review Cache

Represents normalized review data for one place.

Fields may include:

- `placeId`
- `businessName`
- `rating`
- `reviewCount`
- `reviews`
- `source`
- `fetchedAt`
- `expiresAt`

Durable in v1: yes, stored in KV.

Represents one normalized review from Google.

Fields may include:

- `authorName`
- `authorPhotoUrl`
- `rating`
- `text`
- `relativeTimeDescription`
- `publishedAt`

- `data-layout`
- `data-theme`
- `data-position`
- `data-max-reviews`

Durable in v1: no. Configuration lives in the embed code.

## Theme

Represents a built-in or custom visual style.

Built-in themes:

- default
- modern
- professional
- pastel

Theme values should flow through CSS custom properties.

## Refresh Job

Represents a scheduled or manual update of review data.

Durable in v1: only light status/log information in KV if useful.

The refresh job should not become a complex job queue in v1.

# 7. System Layers

## Actual v1 Flow

```text
Client Website
→ widget.js
→ Widget Config Parser
→ Widget Renderer
→ Reviews API Client
→ Worker API
→ KV Storage
```

Refresh flow:

```text
Scheduled Trigger / Manual Admin Endpoint
→ Auth / Admin Token Check when manual
→ Refresh Service
→ Google Places Adapter
→ Normalization Service
→ KV Storage Service
```

## Widget Layer

The widget layer may:

- parse script `data-*` attributes
- locate mount target
- render flyout, grid, or carousel
- apply theme tokens
- fetch cached review data from the Worker API
- handle CTA button behavior
- handle read-more modal behavior
- handle personalization cookie behavior
- pause flyout rotation on hover or hidden tab

The widget layer must not:

- call Google APIs directly
- contain Google API keys
- depend on WordPress
- hardcode client-specific data except through embed options
- require a frontend framework

## Worker API Layer

The Worker API layer may:

- serve widget JavaScript
- serve CSS/assets if needed
- expose `GET /api/reviews`
- expose `GET /api/health`
- expose admin-protected manual refresh endpoint
- route scheduled events to refresh services
- validate inputs
- return normalized response envelopes

The Worker API layer must not:

- call Google during normal public widget requests
- expose secrets
- leak raw provider errors
- accept unbounded input

## Services Layer

Services own business behavior:

- review cache lookup
- place registry lookup
- refresh orchestration
- Google Places fetch
- normalization
- admin manual refresh
- theme/config defaults

## Storage Layer

Storage functions own KV reads/writes.

Code outside the storage layer should not scatter raw KV key construction everywhere.

## Provider Adapter Layer

Google-specific response handling belongs in a Google Places adapter.

Future review sources should be added through provider adapters, not by rewriting widget rendering logic.

# 9. Request and Data Flows

## Widget Load Flow

```text
Browser loads client page
→ Browser requests https://reviews.marketinghero.net/widget.js
→ Widget script finds its own script tag
→ Widget parses data-* attributes
→ Widget creates mount element or uses custom mount selector
→ Widget requests /api/reviews?placeId=...
→ Worker reads normalized review cache from KV
→ Widget renders selected display mode and theme
```

## Scheduled Refresh Flow

```text
Cloudflare scheduled trigger runs every 24 hours
→ Worker loads active place registry
→ Worker refreshes each active place from Google Places API
→ Google response is normalized
→ Normalized data is written to KV
→ Refresh status/log is updated
```

## Manual Refresh Flow

```text
Admin sends POST /api/admin/refresh-place
→ Worker verifies Authorization Bearer token
→ Worker validates place ID
→ Worker fetches Google data
→ Worker normalizes response
→ Worker writes KV cache
→ Worker returns refresh result
```

## Missing Cache Flow

```text
Widget requests place ID
→ Worker checks KV
→ If cache exists, return data
→ If cache is missing, return structured empty/unavailable response
→ Do not call Google inline for public widget request in v1
```

This avoids visitor-facing latency and quota spikes.

# 10. Authentication and Authorization

## Public Widget Requests

Public requests are unauthenticated.

Allowed public endpoints:

```text
GET /widget.js
GET /api/reviews
GET /api/health
```

Public endpoints must validate input and never expose secrets.

## Admin Manual Refresh

Manual refresh must be protected.

Expected pattern:

```text
POST /api/admin/refresh-place
Authorization: Bearer <ADMIN_REFRESH_TOKEN>
```

The token is stored as a Cloudflare Worker secret.

Do not put admin tokens in URLs.

## Google API Key

v1:

- one platform-owned Google Places API key
- stored as a Cloudflare Worker secret
- never sent to browsers
- never stored in embed code

v2:

- per-client Google Places API key support may be added
- client keys must be stored securely
- client keys must not appear in embed code
- a dashboard or secure config process will be required

# 13. Error Handling Pattern

Use consistent error envelopes.

Recommended error codes:

```text
INVALID_PLACE_ID
REVIEWS_NOT_FOUND
REVIEWS_CACHE_EMPTY
GOOGLE_API_FAILED
GOOGLE_RESPONSE_INVALID
REFRESH_FAILED
UNAUTHORIZED
METHOD_NOT_ALLOWED
INVALID_PAYLOAD
KV_READ_FAILED
KV_WRITE_FAILED
INTERNAL_ERROR
```

Rules:

- Public widget requests receive user-safe messages.
- Provider errors are logged safely but not exposed directly.
- Missing cache should not crash the widget.
- Widget rendering should degrade gracefully.
- Manual refresh errors may be more specific but must still not expose secrets.

# 14. Background Processing and Scheduled Jobs

## Scheduled Refresh

Use Cloudflare scheduled triggers.

Default schedule:

```text
Every 24 hours
```

The scheduled job refreshes all active registered places.

## Normal Widget Requests

Normal widget requests must read from KV only.

They must not call Google directly.

## Manual Refresh

Manual refresh exists in v1 and is admin-protected.

It may call Google directly because it is an intentional admin action, not a public visitor request.

## Retry Strategy

Suggested defaults:

- Google transient failure: retry up to 2 times per place
- invalid place ID: no retry
- invalid API key: no retry loop
- KV write failure: log and return failure
- scheduled refresh should continue to the next place when one place fails

# 19. Performance Strategy

Performance priorities:

- tiny widget script
- no runtime dependencies by default
- KV-only read path for public widget requests
- no Google calls during page view
- minimal DOM work
- avoid layout thrashing
- lazy modal content where practical
- cache-friendly static assets
- graceful failure when reviews are unavailable

Suggested optimizations:

- serve minified widget asset when build tooling exists
- use `Cache-Control` headers carefully
- avoid loading external fonts
- avoid image-heavy rendering unless author photos are intentionally enabled
